What this is

A structured reference for every token in the SupplyHouse design system — Tier 1 primitives, Tier 2 semantic tokens, and the typography roles. Each lives in a single chain: a raw value gains meaning, is shaped into a stateful pattern, and is consumed by a component.

A living document reflecting the architecture as of 2026, including open flags still needing team discussion.

Principles

  • Audit-first — tokens only created when a confirmed consumer existed. No speculative tokens.
  • Decision-surfacing — building the system forces previously silent component-level decisions into the open for team alignment.
  • SupplyHouse-specific — TurboTrack, commerce CTA, B2B role tiers, multi-type accordions do not exist in any reference system. The organization reflects this product, not a template.
  • Migration-friendly — makes sense of what already exists in Figma and production. Design and dev can converge incrementally.

Three-tier architecture

  • T1 Primitives — raw values. orange.500 = #F15C02. Never used by components directly.
  • T2 Semantic — purpose-based tokens. The design-to-dev contract.
  • T3 Component — component-scoped tokens. Started: spacing & size for accordion, modal, tooltip, stepper.

A single primitive change propagates through the chain without a component sweep.

The chain

Three tiers — primitive, semantic, component

Read left → right across three tiers: a primitive (①) gains meaning as a semantic token (②), then becomes a component token (③) and is used by components (④). Tier 2 has two layers inside it — Foundation / Identity and an optional Pattern layer. Pattern stays dimmed until a token actually routes through it: click anything to trace its chain and the layers it touches light up.

Field guide

What's in each column, and why

①Tier 1 · Primitives

The raw options — every color, spacing, stroke and effect value, named by ramp (gray.800, blue.700). Components never use these directly.

Why: one place to define the palette — change a primitive and everything downstream follows.

②Tier 2 · Semantic two layers

Foundation / Identity — what a value means, by group:

  • Foundations — text, bg, border, icon, focus
  • Identities — commerce (orange CTA), primary (blue CTA)
  • Communication — feedback, status
  • Domain — fulfillment, role, invite (SupplyHouse-specific)
  • Stroke · Effects — widths, focus rings, shadows

Pattern — a shared, multi-state recipe several components reuse:

text (link)form-fieldform-controlaction-buttonmenu-itemnavigationselectableselectable-cardremovabletoggleswitchbreadcrumb

Why the groups: named by purpose, not appearance. Why .pattern.: it marks the shared-recipe layer so its names don't collide with foundations or components.

③Tier 3 · Component

Tokens scoped to a single component, for values nothing else shares (accordion, modal, tooltip, stepper).

Why: a home for the genuinely one-off, so it doesn't pollute the shared layers.

④ Usage isn't a tier — it lists the components that consume each chain.

Why the names: every token reads {type}.{group}.{variant}.{state}.{slot} — e.g. color.pattern.form-field.error.border — drawn from a fixed vocabulary, so a token's job is legible from its name alone.

Follow a token from raw value to the screen

A token is just a raw value — a color or a number — that gets a name, picks up meaning, and is handed to a component. Below are two real journeys, read left → right (the same order as the columns below). Watch the middle step: there are two valid routes, and every token takes one or the other.

The accordion adds its own Tier-3 token for a component-specific value and skips the Pattern layer. Same destination, different middle step.

Two routes, one rule. The middle step is either a Pattern (shared across components) or a Tier-3 token (specific to one) — never both. Either way the raw value still lives in exactly one place: change orange.500 once, and every purchase button follows.

Type roles

Typography — specimen

Composite text styles, WEB scale. Variants: normal · bold · bold-tight (Body1) · bold-caps (uppercase). Each sample shows real size · weight · line-height · case; the face falls back to Inter where Maison Neue isn't installed. All chips chain to Tier 1.

Foundation template

Responsive type scale

How each role sizes across breakpoints. A proposed, designer-overridable ramp: display/headline compress toward small screens; body & caption stay constant for readability. Line-height is a percentage, so it holds at every breakpoint.

B1 equals the current static scale, so desktop is unchanged — this only adds smaller-breakpoint behaviour. Open: the 1200–1399 band is unassigned (recommend folding into B1), and the numbers are a starting point to agree on.

Foundation · effects

Shadows & effects

The effect styles from the library, mapped to proposed semantic tokens. What reads as a button “border” is one shared effect — effect.button-edge — a black inner shadow that darkens any fill; there is no border token. Two Figma styles share the name Elevation/-2y (card vs. button) and need disambiguating.
